# VeriPost GOALS.md Verification Loop

Started: 2026-05-26T11:44:25-04:00
Updated: 2026-06-01T12:04:30Z

Objective: keep VeriPost aligned with the five `GOALS.md` stories while minimizing the proof and posting flow.

## Current Contract

1. Load app.
2. Enter text to post in World App.
3. Press Post.
4. Use the logged-in World ID inside World App through account context already exposed by the host, without launching wallet-auth, an external connector, fallback wallet, IDKit prompt, or browser authentication redirect.
5. Post tweet with verified human proof.

## Current Source Checks

- The primary Post path requires World App configuration, proof storage configuration, and logged-in World App account context before proof submission.
- In a World App surface, the Post button remains available while account context is pending; production startup passes the build-time `NEXT_PUBLIC_WORLD_APP_ID` into `primeWorldAppRuntime(BUILD_TIME_WORLD_APP_ID)` so official MiniKit account state can initialize with `MiniKit.install(appId)` as soon as `window.WorldApp` is present, before `/api/config` returns, and tapping Post refreshes runtime context with the config app id. If a build-time app id is unavailable, app-id-less official initialization is still gated to a real `window.WorldApp` surface before the config-aware retry. The runtime can send the same narrow World App `command:"init"` state request when account context is still missing, waits in place for account context, and does not call wallet-auth. The raw init fallback can run when a real World App native bridge plus either World App user-agent evidence or strong MiniKit/WorldApp bridge evidence appears before `window.WorldApp`, after native transport appears following an initial missing-transport pass, or when official MiniKit installs but cannot reach the actual native bridge exposed by the host, such as alternate iOS handler casing or Android pages with a `webkit` object. Init retries remain on a one-second cooldown up to five attempts while account context remains missing, and generic ReactNative-style bridge-only pages still receive no native message.
- The client does not call `MiniKit.walletAuth`, `requestNativeWorldIdProof`, `POST /api/world-wallet-auth/nonce`, `POST /api/world-proof/request`, IDKit, sign-message, sign-typed-data, connector, open-url, fallback wallet, X auth, or redirect commands from compose. `MiniKit.install(appId)` is allowed only for official in-app account-state hydration.
- After state-only MiniKit initialization, redirect-capable MiniKit command methods exposed on `window.MiniKit` or older nested command stubs are patched to fail closed and emit a redacted `world_external_navigation_blocked` diagnostic (`minikit-command=blocked`) instead of sending wallet-auth/sign/verify/open-url commands from stale injected handlers.
- On real World App surfaces, known iOS and Android native bridge `postMessage` targets are also guarded directly before and after React hydration: the narrow account-state `command:"init"` payload remains allowed, while stale direct native command payloads such as wallet-auth/sign/open-url are dropped and emit a redacted `world_external_navigation_blocked` diagnostic (`native-command=blocked` or `early-native-command=blocked`). The head guard refreshes briefly during startup so native bridge targets injected after the initial head script still get wrapped before stale pre-hydration handlers can use them.
- The runtime imports `@worldcoin/minikit-js` only to run `MiniKit.install(appId)` for account-state hydration, with app-id-less startup initialization allowed only on real `window.WorldApp` surfaces when no build-time app id is available; it does not call SDK wallet-auth/sign/verify/payment/connector commands from the active proof path.
- The proof request submits `{ draftText, worldAppAccount }` to `/api/proofs` with `x-veripost-world-app-flow: account-context` and the redacted runtime session id.
- The backend validates same-origin/in-app provenance, validates the draft text, verifies the submitted account address through the World ID Address Book, derives the nullifier from the verified account/action/environment/text signal, then stores the proof.
- The client reads World App runtime state and records redacted diagnostics so real-device runs show whether World App transport and account context are present.
- When a World App surface is present but account context is still missing, the client records redacted `world_account_context_pending` diagnostics keyed by runtime shape/init state so short real-device attempts expose the host state before a timeout or page exit.
- Hydrated client runtime diagnostics use `navigator.sendBeacon` first and a same-origin `fetch(..., { keepalive: true })` fallback, matching the pre-hydration guard behavior so `pagehide` / hidden-webview evidence has a better chance of reaching production logs.
- The runtime reads direct and nested `addresses` account arrays and array-wrapped passive account messages, matching the current account-context capture strategy without adding auth commands.
- Legacy redirect-capable auth paths remain trapped. Stale wallet-auth/proof-session/login/sign-in/log-in/world-auth/OAuth/SIWE and common MiniKit/SIWE sample paths return same-origin `410 legacy_auth_removed` for fetches; World App document launches, including no-query/sparse stale auth and exact mini app shell paths, use same-origin `static-fetch` recovery that cleans the visible URL to `/` and fetches the clean root app shell without embedding callback/OAuth target values or serializing stale Next route state, and non-World-App document launches use manual static same-origin recovery HTML.
- Stale root-query and hash handoffs are blocked across common redirect aliases including `callback`, `callbackUrl`, `redirect_uri`, `redirect_to`, `returnTo`, `returnUrl`, `continue`, `target`, `destinationUrl`, `state`, `path`, `pathname`, `next`, and `url`; unknown query keys are also blocked when their values look like URL/path handoffs to stale same-origin auth paths, schemeless X/World/auth hosts, relative `api/auth` or `oauth2` paths, World deep links, World verifier/mini-app URLs, X OAuth, or other off-origin non-tweet targets. The client guards also treat bare `key=value` fragments such as `#fallback=api/auth/signin/twitter` as root handoff candidates instead of letting nested OAuth query strings hide the real handoff value. Safe non-URL values, same-origin `/` and `/proof/*` targets, plus explicit post-proof X/Twitter tweet intents remain allowed.
- The head-installed early navigation guard normalizes stale auth, mini app shell, root-query, and hash handoff URLs to `/` before React hydration, so a stale World App launch path does not remain visible while the compose flow boots.
- The early and hydrated navigation guards also block stale `history.pushState` / `history.replaceState` handoffs and recover `hashchange` / `popstate` stale handoff URLs after load, preserving safe app paths such as `/proof/*` while stripping redirect-capable fragments and root handoff parameters.
- Stale shortcut paths such as `/api/idkit`, `/api/rp-signature`, `/api/sign`, `/api/world-id/*`, `/api/world-wallet-auth/*`, and world-miniapp IDKit/RP/auth variants remain trapped server-side and client-side.
- The app intentionally does not use X login, Auth.js, MiniKitProvider, MiniKit wallet-auth, MiniKit sign-message, MiniKit sign-typed-data, Wagmi fallback, `ethereum.request`, wallet permission APIs, `window.open`, app-owned connector navigation, or app-owned `window.location.assign` for proof creation.
- Browser-level navigation policy allows only same-origin, mail, and explicit `/intent/tweet` navigation on `x.com`/`twitter.com`; it does not broadly allow X OAuth paths.
- Client startup purges stale auth storage/cookies, CacheStorage entries, and service worker registrations. `/sw.js` and `/service-worker.js` are self-unregistering cache-delete scripts for common stale service-worker paths and sanitize stale auth/root/hash client URLs to `/` before navigating controlled clients during activation.
